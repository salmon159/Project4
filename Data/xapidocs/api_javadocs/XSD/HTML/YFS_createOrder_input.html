This IncludeInChainedOrder transaction changes the status of that quantity of the existing line which is equal to the OrderedQty of the child line. If the parent line does not have sufficient quantity to chain from, error is thrown.

DerivedFrom Associated With Table: YFS_ORDER_LINE
Use this element to derive a new OrderLine from an existing OrderLine. If the passed order line is a bundle parent then all the PROD components that have specified KitQty are also pulled in the derived order. If components do not meet status criteria then error is thrown. Bundle-child relationship is maintained in the derived order if parent and components exist in the same derived order.
The existing OrderLine can be identified by using one of the following combinations:
-OrderLineKey
-OrderHeaderKey, PrimeLineNo, and SubLineNo
-OrderNo, DocumentType and EnterpriseCode, PrimeLineNo, and SubLineNo

If the parent line cannot be identified, an error is thrown.

To select only the part of the existing OrderLine associated with a particular release, to derive a new OrderLine from, additional data is needed. In addition to the information to identify the existing OrderLine, supply either the OrderReleaseKey or the ReleaseNo that identifies the release.

Information is taken from the parent line, based on the DerivedOrderDefaultTemplate, stored in the YFS_DOCUMENT_TEMPLATE table. The DerivedOrderDefaultTemplate is chosen on the basis of the DocumentType of the child order.

If there are attributes passed in the createOrder input xml, the values in the passed attributes are used creating the new OrderLine. If passed line is a bundle parent line, derived lines are created for all the eligible components existing in the parent order. All the attributes which are passed to OrderLine and its children elements are applied to all the components that are created as part of creating derived line from a bundle parent. If KitCode and Item attributes are changed while creating the derived line from bundle parent, result may be unpredictable.
Schedule information is never propagated from the parent line to the child line.

The DerivedFromOrderLineKey, DerivedFromOrderHeaderKey and, if applicable, DerivedFromOrderReleaseKey are set on the new OrderLine. The new OrderLineis validated, like all the other lines of the new order.
A transaction is triggered. The transaction is based on the existing order's ProcessType, and the document type of the new order. This IncludeInDerivedOrder transaction changes the status of that quantity of the existing line which is equal to the OrderedQty of the child line. If the parent line does not have sufficient quantity to derive from, an error is thrown.

The Buyer can also specify the ChargeSequence of the PaymentMethod in the createOrder input xml. If a charge sequence is not specified by the Buyer, the Seller's ChargeSequence is followed.

If both the Buyer and the Seller supply ChargeSequence , charges are made in the order that the Seller specifies, but within a particular charge type, such as CreditCard, the Buyer's sequence is honored.

A seller specifies the ChargeSequence as 0 for CUSTOMER_ACCOUNT, 1 for CREDIT_CARD, 2 for OTHER.

The payment methods are charged in the following sequence:
CUSTOMER_ACCOUNT, MyBank-CreditCard, MyOther-CreditCard.
The Customer account is charged because that is the seller's preferred method, and the two credit cards are charged based on the order that the buyer specified.
