Entity Definition
YFS_PAYMENT_TYPE
( CONFIGURATION Table )
This table stores all payment types supported by various organizations.



You can extend this table by adding columns but not by creating hang-off table(s). Please refer to the Customization Guide for detailed procedures and restrictions.

Tables should be accessed only through exposed APIs and Business Intelligence. Accessing them directly may result in issues during application of hot-fixes, service packs, and during upgrade to future releases of applications.

Columns
Column NameData TypeDescription
PAYMENT_TYPE_KEYChar (24)The primary key for the YFS_PAYMENT_TYPE table.  
PAYMENT_TYPEVarchar (20)The unique identifier for the payment type. 
ORGANIZATION_CODEChar (24)The organization that is associated with this payment type.
A logical foreign key to the following table:
  • YFS_ORGANIZATION 
PAYMENT_TYPE_DESCRIPTIONVarchar (100)The description of the payment type. 
PAYMENT_TYPE_GROUPVarchar (40)This filed indicates the payment type group, which will determine behavior. The valid values are:
  • CREDIT_CARD: Indicates that the payment type is in the credit card group.
  • CUSTOMER_ACCOUNT: Indicates that the payment type is in the customer account group.
  • STORED_VALUE_CARD: Indicates that the payment type is in the stored value card group.
  • OTHER: Indicates that the payment type is in the other group. Payment types such as checks can be configured in this group.
  • DEBIT_CARD: Indicates that the payment type is in the debit card group. 
INTERFACE_TIMEVarchar (40)This field is not currently used. 
COLLECT_EXTERNAL_THROUGH_ARChar (1)This field is not currently used. 
FINANCIAL_ORG_TYPEVarchar (20)The type of the financial organization. 
FINANCIAL_ORG_CODEChar (24)The organization code against which financial transactions are recorded. 
MERCHANT_IDVarchar (40)The identifier of the merchant used for the collection. 
FINANCIAL_HANDLING_REQUIREDChar (1)This flag indicates whether financial handling is required. 
CHARGE_SEQUENCENumber (5,0)The sequence in which charges needs to be collected if multiple payment types are supported by an organization. 
CHARGE_INSTEAD_OF_AUTHChar (1)This flag indicates if the requestCollection API should create a CHARGE request against this Payment Type in all scenarios where an AUTHORIZATION request would be created. 
CHARGE_UP_TO_AVAILABLEChar (1)This flag indicates whether the user exit getFundsAvailable should be invoked to determine the funds available on this payment account. 
CHARGE_CONSOL_ALLOWEDChar (1)This flag indicates whether charge consolidation is allowed for a payment type. 
MIN_WAIT_TIME_FOR_CONSOLNumber (14,4)The minimum time to wait for utilizing the opportunity to consolidate charges. 
VALID_FOR_RETURNChar (1)This flag indicates whether this payment type is valid for returns. 
REFUND_SEQUENCENumber (5,0)The sequence in which charges need to to be refunded if multiple payment types are supported by an organization. 
DEFAULT_FOR_RETURNChar (1)This flag indicates the usage of a payment type as the default one while processing returned goods. 
LINK_REFUND_TO_SETTLEMENTChar (1)This flag indicates whether the refunds should be linked to settlement charge transaction for this payment type. This flag will come into action for only when refund configuratin is refund to same payment method. 
REFUND_SAME_ACCOUNTChar (1)This flag indicates whether refunds should be issued to the same account on which the payment was made. 
NEW_REFUND_PAYMENT_TYPEVarchar (20)The payment type that needs to be used for refunds and is only set if not refunding to the same account. Payment type selection is valid only if it belongs to payment type group STORED_VALUE_CARD or OTHER. 
HAS_ALT_REFUND_CONSTRAINTChar (1)This flag indicates whether this payment type has some alternate refund payment type to use in case the refund amount meets the constraint. 
REFUND_CONSTRAINT_OPERATORChar (4)The operator to be used in evaluating the refund amount against the Refund Constraint Value. The valid values are:
  • GT: Indicates that the operator is greater than.
  • LT: Indicates that the operator is less than. 
REFUND_CONSTRAINT_VALUENumber (15,2)The value that will be used to compare the refund amount against. 
REFUND_CONSTRAINT_PAYMENT_TYPEVarchar (20)The payment type that needs to be used instead if the refund amount meets the constraint formed by the refund constraint operator and value when a new payment is created for refund purposes. The payment type selection is valid only if it belongs to payment type group STORED_VALUE_CARD or OTHER. 
REFUND_REQUIRES_FULFILLMENTChar (1)This flag indicates whether a new order needs to be created for the Refund Item ID and UOM when a refund needs to be issued on a new payment account for this payment type. 
REFUND_ITEM_IDChar (40)The item to be included in a new refund fulfillment order when a refund is issued to this payment type. 
REFUND_ITEM_UOMVarchar (40)The unit of measure for the refund item. 
AWAIT_INFO_ON_NEW_REFUNDChar (1)This flag indicates whether payment information is required before processing a payment method that is newly created for refunds. 
AUTH_REVERSAL_STRATEGYVarchar (20)This field controls when reverse authorization requests will be created for this payment type. The valid values are:
  • NO_REVERSE: This strategy will never generate reversal requests for authorizations. This is the default strategy.
  • EXPIRED: This strategy will generate a reversal request for an authorization after it expires.
  • REVERSE_EXCESS: This strategy will still reverse authorizations that are expired, but it will also reverse authorizations that are in excess of either the order total or the CTR they are mapped to. In additon, any authorization that cannot be mapped to a CTR and is marked as unusable will also be reversed. 
ALLOW_AUTH_TO_EXCEED_SETTLE_RQChar (1)This rule determines whether authorizations larger than the settlement amount are able to be used for settlement for this payment type. If IS_MULTIPLE_REQ_ALLOWED or CHARGE_CONSOL_ALLOWED are Y, this is implicitly Y. 
AUTH_EXPIRATION_HOURSNumber (7,0)Number of hours an authorization is valid before expiring for this payment type. This value is not used internally by the system, but may be referenced for the computation of the AuthorizationExpirationDate when processing payment transactions. 
AUTH_REVERSAL_HOURSNumber (7,0)Number of hours an authorization is valid for reversal for this payment type. This value is not used internally by the system and serves as an informational field. 
PARTIAL_REVERSAL_SUPPORTChar (1)This rule determines whether reverse authorization requests support partial reversal for this payment type. If Y, partial reversal is supported. If N, only full reversal is supported. 
ZERO_AMOUNT_AUTH_SUPPORTEDChar (1)This flag indicates whether or not the system will attempt a zero authorization when this Payment Type is added on the Order. 
NO_PROCESSING_REQUIREDChar (1)This flag indicates that when the Payment Type is added on the Order, it is already processed. Examples of this Payment Type are checks/cash. A pre processed Charge Transaction record of type CHARGE is created for this Payment Type. 
VOIDABLEChar (1)This flag indicates that the Payment Type allows a void operation. 
CASH_BACK_LIMITNumber (15,2)The maximum amount that can be given back as cash for payment methods associated with this payment type. 
PAYMENT_TEMPLATE_TYPEVarchar (100)This field indicates User Interfaces to use the template required to capture Payment Methods of this Payment Type. 
REV_AUTH_WHEN_SUSPENDChar (1)Reverse unused authorizations when the payment method is suspended. Currently only valid for payment method of group type CUSTOMER_ACCOUNT.

MODIFYUSERIDVarchar (40)User who modified the record. 
CREATEPROGIDVarchar (40)Program that created the record. 
MODIFYPROGIDVarchar (40)Program that modified the record. 
LOCKIDNumber (5,0)An integer used to track and prevent concurrent modifications.
